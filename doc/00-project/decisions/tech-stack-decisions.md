## ADR-001: 前端技术栈选型

### 背景

项目需要支持 Web网站、管理后台、微信小程序三个客户端，需要选择合适的前端技术栈。

### 考虑因素

1. **开发效率**: 团队熟悉度、生态成熟度
2. **性能表现**: 首屏加载、运行时性能
3. **SEO支持**: 搜索引擎优化需求
4. **跨端能力**: 一套代码多端运行
5. **长期维护**: 社区活跃度、版本更新

### 备选方案

#### 方案A: Vue 3 全家桶
- **技术组合**: Vue 3 + Vite + Pinia + Element Plus
- **优点**: 
  - 国内生态成熟，文档中文友好
  - 学习曲线平缓
  - 性能优秀
- **缺点**:
  - SSR方案(Nuxt)不如Next.js成熟
  - 企业级组件库不如React丰富

#### 方案B: React + Next.js
- **技术组合**: React 18 + Next.js 14 + Zustand + Ant Design
- **优点**:
  - Next.js SSR/SSG能力业界领先
  - React生态最丰富
  - 企业级组件库(Ant Design)完善
  - 招聘更容易
- **缺点**:
  - 学习曲线稍陡
  - 国内文档相对较少

#### 方案C: Angular
- **优点**: 企业级框架，功能完整
- **缺点**: 过于重型，学习成本高，国内使用减少

### 决策结果

**选择方案B: React + Next.js**

**理由**:
1. Next.js的SSR能力对SEO至关重要
2. 产品面向B端和企业用户，Ant Design更合适
3. React招聘市场更大，团队扩展容易
4. 小程序使用Taro(React语法)，技术栈统一

### 实施计划

- Web网站: Next.js 14 (App Router)
- 管理后台: React 18 + Ant Design Pro
- 小程序: Taro 3 + React

## ADR-002: 后端服务分层架构

### 背景

PDF处理是CPU密集型任务，需要设计合理的后端架构平衡开发效率和执行性能。

### 考虑因素

1. **开发效率**: 业务逻辑快速迭代
2. **执行性能**: PDF转换处理速度
3. **资源利用**: 服务器成本控制
4. **维护成本**: 团队技术栈统一性

### 备选方案

#### 方案A: 纯Node.js架构
- **技术组合**: NestJS + 子进程调用转换工具
- **优点**: 
  - 技术栈统一，全TypeScript
  - 开发效率高
- **缺点**:
  - Node.js不适合CPU密集型任务
  - PDF处理会阻塞事件循环
  - 内存占用较高

#### 方案B: 纯Go架构
- **技术组合**: Go + Gin/Echo
- **优点**:
  - 性能最优
  - 并发处理能力强
  - 资源占用低
- **缺点**:
  - 开发效率不如Node.js
  - 生态不如Node.js丰富
  - 团队学习成本

#### 方案C: 混合架构 (Node.js + Go)
- **API层**: Node.js (NestJS) - 处理业务逻辑
- **处理层**: Go (Gin) - 处理PDF转换
- **通信**: HTTP/gRPC + 消息队列
- **优点**:
  - 兼顾开发效率和执行性能
  - API层快速迭代
  - 处理层高性能
  - 可独立扩展
- **缺点**:
  - 技术栈增加
  - 系统复杂度提高

### 决策结果

**选择方案C: 混合架构 (Node.js + Go)**

**理由**:
1. PDF处理是CPU密集型，Go的goroutine更适合
2. 业务API需要快速迭代，Node.js开发效率高
3. 两层可独立扩展，资源利用更合理
4. 业界有成功先例(如石墨文档)

### 架构分层

```
┌─────────────────────────────────────┐
│         API Gateway (Nginx)         │
└─────────────────────────────────────┘
                   │
       ┌───────────┴───────────┐
       ▼                       ▼
┌─────────────┐         ┌─────────────┐
│  业务服务    │         │  处理服务    │
│  (Node.js)  │◄───────►│    (Go)     │
│             │  gRPC   │             │
│ • 用户管理   │         │ • PDF转换    │
│ • 订单支付   │         │ • 文件处理   │
│ • 任务调度   │         │ • OCR识别    │
└─────────────┘         └─────────────┘
```

## ADR-003: 数据库选型

### 背景

需要选择主数据库存储用户、订单、任务等业务数据。

### 考虑因素

1. **数据一致性**: 事务支持
2. **查询能力**: 复杂查询、JSON支持
3. **扩展性**: 读写分离、分库分表
4. **运维成本**: 备份、监控、优化

### 备选方案

#### 方案A: MySQL 8.0
- **优点**: 国内使用最广泛，运维成熟
- **缺点**: JSON功能较弱，复杂查询性能一般

#### 方案B: PostgreSQL 15
- **优点**:
  - JSONB支持优秀，适合灵活数据
  - 复杂查询性能更好
  - 功能更丰富(窗口函数、CTE等)
  - 开源协议更友好
- **缺点**: 国内DBA相对较少

#### 方案C: MongoDB
- **优点**:  Schema灵活
- **缺点**: 事务支持较晚，复杂查询性能差

### 决策结果

**选择方案B: PostgreSQL 15**

**理由**:
1. JSONB支持适合存储任务配置、文件元数据
2. 复杂查询性能优于MySQL
3. 功能丰富，减少应用层复杂度
4. 开源友好，无商业风险

## ADR-004: PDF转换引擎选型

### 背景

PDF转换是核心功能，需要选择可靠的转换方案。

### 考虑因素

1. **转换质量**: 格式保留完整性
2. **支持格式**: 覆盖需求文档中的格式
3. **性能**: 转换速度
4. **成本**: 授权费用 vs 自研成本
5. **可控性**: 私有化部署需求

### 备选方案

#### 方案A: 纯开源方案
- **工具**: LibreOffice + Apache PDFBox + Tabula
- **优点**: 零授权成本，完全可控
- **缺点**: 
  - 转换质量不如商业方案
  - 复杂文档格式丢失
  - 维护成本高

#### 方案B: 纯商业方案
- **工具**: Aspose.PDF (Java/.NET)
- **优点**: 转换质量最好
- **缺点**:
  - 授权费用高(年付$4000+)
  - 依赖Java运行时
  - 有授权验证风险

#### 方案C: 混合方案 (推荐)
- **策略**: 开源为主，商业兜底
- **实现**:
  - 常规转换: LibreOffice
  - 质量不佳时: 调用商业API
  - 复杂表格: Tabula + 自定义算法
- **优点**:
  - 平衡成本和质量
  - 逐步优化自研能力
  - 降低商业依赖

### 决策结果

**选择方案C: 混合方案**

**实施策略**:
1. **第一阶段**: LibreOffice + PDFBox 实现MVP
2. **第二阶段**: 引入Aspose作为质量兜底
3. **第三阶段**: 基于开源组件自研优化

## ADR-005: 消息队列选型

### 背景

需要消息队列实现异步任务处理和系统解耦。

### 考虑因素

1. **可靠性**: 消息不丢失
2. **延迟**: 任务处理延迟
3. **功能**: 延迟队列、死信队列等
4. **运维**: 部署复杂度

### 备选方案

#### 方案A: Redis + BullMQ
- **优点**: 部署简单，与现有Redis复用
- **缺点**: 功能简单，不适合复杂路由

#### 方案B: RabbitMQ
- **优点**:
  - 功能丰富(路由、延迟、死信)
  - 可靠性高
  - 管理界面友好
- **缺点**: 性能不如Kafka

#### 方案C: Kafka
- **优点**: 吞吐量最高
- **缺点**: 运维复杂，延迟较高

### 决策结果

**选择: Redis + BullMQ + RabbitMQ 组合**

**分工**:
- **BullMQ**: 任务队列(转换任务调度)
- **RabbitMQ**: 事件总线(服务解耦、通知)

**理由**:
1. BullMQ基于Redis，部署简单，适合任务队列
2. RabbitMQ功能丰富，适合复杂事件路由
3. 两者互补，不重复造轮子

## ADR-006: 部署架构选型

### 背景

需要选择适合项目阶段的部署方案。

### 考虑因素

1. **成本**: 服务器和运维成本
2. **弹性**: 应对流量波动
3. **可控性**: 私有化部署需求
4. **团队**: 运维能力

### 备选方案

#### 方案A: 传统服务器部署
- **优点**: 简单直接，成本低
- **缺点**: 扩展性差，运维繁琐

#### 方案B: Docker + Docker Compose
- **优点**: 环境一致性，部署简单
- **缺点**: 无自动扩缩容

#### 方案C: Kubernetes
- **优点**:
  - 自动扩缩容
  - 服务发现和负载均衡
  - 滚动更新
  - 资源利用率高
- **缺点**:
  - 学习成本高
  - 运维复杂

### 决策结果

**分阶段选择**:

| 阶段 | 方案 | 说明 |
|------|------|------|
| V1.0 MVP | Docker Compose | 快速上线，单服务器部署 |
| V2.0 增长 | Kubernetes | 需要弹性伸缩 |
| V3.0 企业版 | K8s + 私有化 | 支持私有化部署 |

## ADR-007: 云服务选型

### 背景

需要选择云服务商和具体产品。

### 考虑因素

1. **价格**: 服务器、带宽、存储成本
2. **稳定性**: SLA保障
3. **产品丰富度**: 是否需要PaaS产品
4. **合规**: 数据本地化要求

### 备选方案

#### 方案A: 阿里云
- **优点**: 产品最全，文档完善
- **缺点**: 价格相对较高

#### 方案B: 腾讯云
- **优点**: 价格有优势，社交产品集成好
- **缺点**: 部分产品不如阿里云成熟

#### 方案C: AWS/ Azure
- **优点**: 技术领先
- **缺点**: 国内访问慢，合规复杂

### 决策结果

**选择: 阿里云为主，腾讯云为辅**

**分工**:
- **阿里云**: 服务器ECS、RDS、OSS、CDN
- **腾讯云**: 微信小程序相关服务

**理由**:
1. 阿里云产品成熟，文档完善
2. 国内用户访问速度快
3. 满足数据本地化要求
