# PDF合并 MVP 交互设计说明

## 1. 目标

围绕 `PDF合并` MVP，确保用户可以在最短路径内完成：
1. 添加至少 2 个有效 PDF
2. 调整顺序
3. 发起合并
4. 下载结果

交互原则：明确状态、实时反馈、可恢复错误。

---

## 2. 页面状态机

## 2.1 状态定义
- `empty`：空状态，未满足最小文件数
- `editing`：列表编辑中，可上传/排序/删除
- `processing`：上传或合并处理中
- `success`：合并成功，可下载
- `error`：任务失败，可重试或清空

## 2.2 转换规则
- `empty -> editing`：用户成功添加首个文件
- `editing -> processing`：点击开始合并且校验通过
- `processing -> success`：任务完成
- `processing -> error`：任务失败或超时
- `error -> editing`：用户重试并修正文件

---

## 3. 关键交互流程

## 3.1 文件添加
- 支持点击选择与拖拽批量导入
- 同时添加时按进入列表顺序落位
- 上传前先做前端校验：格式、单文件大小、总大小、数量

失败反馈：
- 非 PDF：标记行为错误，提示仅支持 PDF
- 超限：提示上限值与建议动作（压缩/分批）

## 3.2 列表排序
- 主交互：拖拽整行调整
- 备用交互：上移/下移按钮（键盘友好）
- 排序后立即保存当前顺序用于提交

## 3.3 发起合并
- 按钮可用条件：就绪文件数 >= 2 且无上传失败项
- 点击后进入 `processing`，禁用重复提交
- 文案分阶段展示：
  - 正在上传文件
  - 正在合并文件
  - 正在生成下载文件

## 3.4 成功与失败
- 成功：突出下载按钮，其次“再合并一组”
- 失败：展示错误标题 + 原因 + 下一步（重试/替换文件）
- 保留列表上下文，避免用户从头操作

---

## 4. 交互细节规范

## 4.1 按钮状态
- 默认：可点击
- 悬停：颜色加深，不缩放
- 加载：显示 spinner + 禁用
- 禁用：低对比样式并保留说明文案

## 4.2 进度反馈
- 行级上传进度：百分比 + 条形
- 任务级进度：阶段文本 + 全局进度条
- 若存在排队，显示“前方等待 x 项”或“预计等待 y 秒”

## 4.3 错误恢复
- 行级错误提供 `重试上传`
- 任务级错误提供 `重试合并`
- 连续失败 2 次后提示反馈入口

## 4.4 可访问性
- 错误消息使用 `role="alert"` 或 `aria-live`
- 拖拽排序功能必须有键盘替代方式
- 焦点顺序：上传区 -> 文件列表 -> 参数区 -> 主按钮

---

## 5. 文案建议

主按钮：
- 初始：`开始合并`
- 处理中：`正在合并...`
- 成功后辅助：`再合并一组`

提示文案：
- 空状态：`至少添加 2 个 PDF 文件后可开始合并`
- 超限：`总大小超过上限，请减少文件或升级会员`
- 失败：`合并失败，请重试；若持续失败请更换文件`

---

## 6. 埋点建议（用于体验优化）

- `merge_file_add`
- `merge_order_change`
- `merge_submit`
- `merge_success`
- `merge_fail`
- `merge_download`

建议附带字段：
- 文件数量
- 总大小
- 错误码
- 总耗时

- [ ] 按钮禁用逻辑与校验规则一致
- [ ] 所有失败场景有明确下一步动作
- [ ] 键盘可操作关键流程
- [ ] 长时任务中无“静默等待”状态
