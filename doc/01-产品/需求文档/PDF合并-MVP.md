## 1. MVP 范围摘要

| 能力 | MVP 是否包含 |
|------|----------------|
| 多文件上传 | 是 |
| 列表内排序（整文件顺序） | 是 |
| 整文件合并（每文件全部页面按序追加） | 是 |
| 合并后下载单个 PDF | 是 |
| 基础校验（仅 PDF、单文件大小、总个数/总大小上限） | 是 |
| 失败明确文案与可执行建议 | 是 |
| 上传与合并过程进度反馈 | 是 |
| 匿名使用或简单登录（可选） | 是 |

**限额占位符**：具体数值在「主数据表」冻结前使用 `N_files`（单次最多文件数）、`S_single`（单文件上限）、`S_total`（单次总大小上限）。开发前须与 [PDF工具集-PRD.md](./PDF工具集-PRD.md) 中免费/会员规则对齐为唯一真值。

**本期不做**：单文件内选页、书签、统一页面尺寸、页码连续选项、缩略图预览、OCR/Office 互转等与合并 MVP 无关的能力。

## 2. 用户流程（Happy Path）

```mermaid
flowchart LR
  enter[进入合并页]
  add[添加PDF到列表]
  order[调整顺序]
  validate[客户端校验通过]
  submit[点击开始合并]
  queue[服务端入队处理]
  done[结果页下载]
  enter --> add --> order --> validate --> submit --> queue --> done
```

- **最小步数**：进入页 → 至少 2 个有效 PDF → 排序（可选，默认上传序）→ 开始合并 → 下载（5 步内完成认知闭环）。
- **合并顺序**：结果 PDF 的页面顺序 = 列表 **从上到下**（或列表渲染顺序）；每份源文件 **全文所有页** 按序追加，不做跨文件插页。

## 3. 页面信息架构与状态

| 状态 | 界面要点 | 主按钮 |
|------|----------|--------|
| **空状态** | 说明「至少 2 个 PDF」、拖拽区、选文件按钮；可放格式与大小提示（引用限额） | 「选择文件」可用；「开始合并」禁用并说明原因 |
| **列表编辑** | 已选文件行：图标/原名、大小、单文件状态（等待上传/上传中/已就绪/失败）、移除 | 「添加文件」；「开始合并」仅当就绪文件数 ≥ 2 且全部成功上传 |
| **合并中** | 全页或底部固定区展示任务进度（见第 5 节）；禁用重复提交；可允许「取消」（若技术可实现则显示，否则 MVP 隐藏取消） | 「开始合并」变为禁用或加载态 |
| **成功** | 输出文件名（默认 `merged.pdf` 或可编辑一行）、文件大小、**主 CTA「下载」**、次 CTA「再合并一组」 | 下载为主 |
| **失败** | 错误标题 + **可执行建议** + 错误码（小字/展开）；保留当前列表以便改文件后重试 | 「重试」复用同一列表（需重新校验）；「清除并重新选择」 |

### 3.1 列表行交互（MVP）

- **排序**：拖拽整行调整顺序；提供 **上移/下移** 作为键盘/无障碍备选（二选一即可写进验收）。
- **删除**：单行移除；删除二次确认可选，默认单次删除无确认以减步骤。
- **不重做**：MVP 不提供单文件内选页、不提供书签/统一页面尺寸。

## 4. 多文件上传

- **入口**：点击选择多文件、`input[multiple]`、拖拽一批文件到拖拽区；重复添加时策略二选一（产品冻结）：**去重（同名同大小）** 或 **允许重复各合并一次**。
- **并发**：建议前端 **限制同时上传数**（如 3），其余排队，避免弱网占满；每行独立 **上传进度条**（百分比或已传/总大小）。
- **断点续传**：MVP **可不做**；若不做，需在文案中提示「请勿刷新页面」，大文件场景由限额与后续版本承接。
- **展示字段**：原始文件名（截断+悬浮全称）、格式化大小、状态图标；上传完成后行内变为 **就绪**（可参与合并）。

---

## 5. 基础校验（客户端先行 + 服务端强校验）

| 校验项 | 客户端 | 服务端 | 失败文案原则 |
|--------|--------|--------|----------------|
| 扩展名/MIME | 必选 `.pdf`（可提示隐藏扩展名用户） | 校验 magic bytes，防改后缀 | 「仅支持 PDF」「文件可能不是有效的 PDF」 |
| 单文件大小 | 上传前拦截 | 再次校验 | 「单个文件超过上限（最大 S_single），请压缩或分批处理」 |
| 文件个数 | 至少 2、至多 N_files | 一致 | 「请至少选择 2 个 PDF」「单次最多合并 N_files 个文件」 |
| 总大小 | 各已就绪文件大小之和 ≤ S_total | 一致 | 「文件总大小超过上限（最大 S_total），请减少文件或升级权益」 |
| 加密 PDF | 上传前未必可知，依赖服务端 | 打开失败时返回明确错误 | 见第 6 节策略 A/B |
| 损坏/无法读取 | — | 打开失败 | 「文件已损坏或无法读取，请更换文件后重试」 |

---

## 6. 加密 PDF 策略（MVP 须二选一写死）

- **策略 A**：不支持 — 失败文案说明「暂不支持已加密 PDF，请先在本机解密后再上传」。
- **策略 B**：支持 — 在失败或预检后引导 **该行输入密码**（每加密文件一个密码字段），再单独重试该文件上传或合并前验证。

---

## 7. 处理中进度与反馈

- **上传阶段**：每文件独立进度；列表顶部可显示「已上传 x / y」。
- **合并阶段**（服务端异步）：用户点击「开始合并」后进入合并中状态，展示 **阶段式文案 + 不确定进度条或百分比**（若引擎可回报页级进度则显示「正在合并第 i / n 个文件」更优）。
- **排队**：若限流，显示 **队列位置** 或「前方等待约 x 项」；避免静默卡住。
- **超时**：超过产品约定时间（如 120s，可配置）展示「处理时间较长，请稍后在历史/结果页查看」——若 MVP 无历史页，则改为「请稍后重试」并提供客服/反馈入口（可选）。

---

## 8. 合并后下载

- **产物**：单个 `.pdf` 文件；默认名 `merged.pdf` 或 `合并_YYYYMMDD_HHmm.pdf`（二选一）。
- **链接**：一次性或短时有效下载链接（具体 TTL 与架构一致，产品对用户说明「建议 24h 内下载」类预期）。
- **安全**：下载前可再次校验登录态/任务归属（若匿名用 session 或设备令牌）。

---

## 9. 匿名与简单登录（可选）

| 模式 | 行为建议 |
|------|----------|
| **匿名** | 基于浏览器会话或匿名令牌绑定任务；刷新丢失会话则 **无法找回同一任务**（文案说明）；限额用 **IP + 指纹/Cookie** 由风控定义，产品只写「每日次数」规则。 |
| **简单登录** | 手机/邮箱/第三方其一；任务与 **用户 ID** 绑定，可在「我的任务」或邮件内找回下载链接（若 MVP 不做任务列表，则登录仅用于额度与风控）。 |

**MVP 推荐**：先 **匿名可用**，登录为 **可选升级**（解锁更高限额或保存记录），减少首访摩擦。

## 10. 失败文案与错误场景（研发可落错误码表）

| 场景 | 用户可见标题 | 说明与下一步 |
|------|--------------|----------------|
| 非 PDF / 伪 PDF | 无法识别文件格式 | 仅支持 PDF；请检查文件扩展名与内容 |
| 单文件超限 | 文件过大 | 给出 S_single；建议压缩或分批 |
| 总个数超限 | 文件过多 | 给出 N_files；建议分多次合并 |
| 总大小超限 | 总容量过大 | 给出 S_total |
| 加密（策略 A） | 不支持加密文件 | 请本地解密后重新上传 |
| 加密密码错误（策略 B） | 密码错误 | 请重新输入该文件密码 |
| 合并引擎失败 | 合并失败 | 请稍后重试；若持续失败请联系反馈 |
| 网络中断 | 上传失败 | 检查网络后点击该文件重试上传 |

## 12. 验收检查清单（产品自测）

- [ ] 2 个最小、N_files 个最大、边界大小 S_single / S_total 均覆盖
- [ ] 拖拽排序后合并顺序与预期一致（可用页码水印测试 PDF）
- [ ] 列表中删除/新增后提交正确
- [ ] 每种错误码对应文案与「下一步」清晰
- [ ] 合并结果用 Adobe / 浏览器 / 至少一款国产阅读器可打开
- [ ] 匿名刷新后的预期行为与文案一致

---

## 13. 非功能与观察指标（建议）

- **质量**：任务成功率 ≥ 98%（排除用户主动取消）；合并结果可被常见阅读器打开。
- **性能**：普通文档（如总页数小于 200、总大小在免费上限内）P95 完成时间可与整体 PRD 非功能目标对齐（可先以 60 秒内为内部基线，再收紧）。
- **体验**：从进入到拿到下载链接/文件的步骤数与失败重试率可追踪。
